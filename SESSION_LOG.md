## 2026-07-20 — MVP-обзор, сверка с эталонными документами заказчика, Этап 6 (экспорт в PDF): требования собраны и согласованы

   `03_План_график_реализации.xlsx` — застейджированы в git (не закоммичены). Текст извлечён через

   Второй `analyst`-агент сверил их с фактической реализацией. **Главный вывод: этот SRS реально

     частично есть, UI нет).

     «после стабилизации основного функционала», но реализован раньше формально более
     приоритетных (Should/Must) пунктов выше — стоит сверить с заказчиком, было ли это осознанной
     сменой порядка.
   - **Сверх документов:** блочная модель слайда (`layoutKind`) не описана в SRS как модель
     данных, но не противоречит ему — SRS §6.2 анализирует ту же референс-презентацию и приходит
     к похожим текстовым выводам, это более глубокая техническая проработка той же идеи.

3. **Этап 6 — экспорт/печать презентации в PDF: сбор требований завершён, реализация ещё не
   начата.** Требования собирал `analyst` (`REQUIREMENTS.md`, запись «Этап 6: сбор требований»).
   Ключевая находка при разборе кода: тип блока `CHART_IMAGE` (из Этапа 5в) сегодня везде пустая
   заглушка — картинки из pptx-импортёра никогда не персистятся, в системе физически нет ни
   одного слайда с реально сохранённой картинкой; PDF-экспорт «финансовых» слайдов с графиками
   будет содержать то же пустое место, что и сейчас в браузере, если объём не расширять отдельно.
   Пользователь ответил на все 7 открытых вопросов:
   - экспортируется вся презентация целиком **и** отдельный слайд (обе опции);
   - инициировать может **и ADMIN, и SPEAKER**;
   - права на экспорт = права на чтение презентации сегодня (без расширения);
   - технический подход — **headless Chromium** (playwright/puppeteer), переиспользующий
     существующую React-вёрстку `/presentation`, точность соответствия браузеру важнее веса
     зависимости — деплой-инструкция потребует нового шага установки браузерного бинарника;
   - хранение — **только генерация по требованию, без сохранения на диске сервера** (файловая
     инфраструктура проекту по-прежнему не нужна);
   - вёрстка — альбомная A4, 1 слайд = 1 страница, без принудительной обрезки контента;
   - нужна титульная страница (цикл/дата экспорта), оглавление не нужно.

   **Важно:** по двум из этих пунктов эталонный SRS предполагает другое — §2.3/FR-EXP-01 говорит
   «экспорт только ADMIN», а НФТ §4.3 называет PDF «дополнительной точкой восстановления», то есть
   требует постоянного хранения на сервере. Пользователь **явно подтвердил, что расхождение по
   обоим пунктам согласовано с заказчиком** — решение сознательное, не по недосмотру. Зафиксировано
   в `REQUIREMENTS.md` (новый подраздел «Ответы на открытые вопросы» в записи Этапа 6).

**Что сделали:** реализовали и проверили Этап 5в по плану `crispy-stirring-cherny.md` (сохранён в

закрыты пользователем: `chart-image` — только результат импортёра (без ручной загрузки где-либо в
этом этапе); гранулярность — админ выбирает один слайд-образец → один шаблон; экран приёма —
предзаполнение существующего конструктора (`TemplatesPage.tsx`, Этап 5б), не новый экран; у 2
существующих легаси dev-шаблонов нет исходного pptx, их миграция вне объёма.

**Находка при подготовке плана:** распаковал `референс общей презентации.pptx` и проверил XML
напрямую — `docs/design-review.md` ошибочно называл графики на слайде 10 «растровыми
embedded-картинками»; на деле это живые OOXML-чарты (`ppt/charts/chart*.xml`), а реальные PNG в
файле относятся к титульному слайду и мелким иконкам. Эвристика учла это: и картинки, и
обнаруженные чарты одинаково предлагаются как `chart-image`-блок, но превью есть только у
картинок (чарт не рендерим).

**Ключевое упрощение объёма (обнаружено при планировании, сообщено пользователю в плане):** раз
`chart-image` — только результат импортёра и никогда не персистится как значение блока, а картинка
нужна лишь для одноразового превью во время интерактивного разбора — **постоянное файловое
хранилище в этом этапе не потребовалось вообще**. Весь pptx обрабатывается в памяти
(`multer.memoryStorage()`), картинки возвращаются как base64 прямо в ответе API. Не пришлось
трогать `deploy/nginx.conf.example`, не появилось директории `uploads/`.

**Backend:**
- Схема: `CHART_IMAGE` добавлен в `enum BlockType` (аддитивная миграция
  `20260719153951_add_chart_image_block_type`), новых моделей нет.
- `server/src/pptx/parse.ts` (новый) — чистые функции без Express/Prisma: `parsePptx` (zip+OOXML
  через `jszip`+`fast-xml-parser`, извлечение текстовых фигур/таблиц/картинок/чартов по позиции),
  `proposeTemplate` (эвристика → `layoutKind`+блоки). Проверено напрямую на реальном референсе —
  совпадает с ручным дизайн-разбором (слайды 7/9/11/14 → `QUADRANT`, слайды с таблицами/чартами →
  `FINANCIAL_CHART`).
- `server/src/routes/pptxImport.ts` (новый) — `POST /api/pptx-import/parse`, ADMIN-only,
  `multer.memoryStorage()`, лимит 20МБ, отдельный 4-аргументный error-мидлварь для чистого `400`
  на `MulterError` (не общий `500`-хендлер `app.ts`).
- `slides.ts`: `defaultValueFor`/`validateBlockValue` — `CHART_IMAGE` всегда `{path: null}` в этом
  этапе, `PATCH` пропускает неизменённое значение (иначе сохранение остальных блоков слайда
  сломалось бы для любого шаблона с chart-image-блоком), но отклоняет непустой `path` (`400`).
- `templates.ts`: свой независимый список `BLOCK_TYPES` тоже дополнен `CHART_IMAGE` — это была
  находка ревью-агента перед реализацией (без неё `POST /api/templates` с импортированным блоком
  падал бы `400` «Неизвестный тип блока», хотя `slides.ts` уже был бы готов).

**Frontend:**
- `client.ts`: `request()` научен работать с `FormData`-телом; новый метод `parsePptx(file)`.
- `slideBlocks.tsx`: `CHART_IMAGE`-кейсы в `BlockPreview`/`BlockEditor`/`isBlockEmpty` (заглушка

- `TemplatesPage.tsx`: кнопка «Импортировать из pptx» + список слайдов-предложений, клик
  предзаполняет форму конструктора Этапа 5б (название — пустое, вводит админ), превью картинки у
  `chart-image`-строк если есть, тип `CHART_IMAGE` в селекторе доступен только для уже
