## 2026-07-18 — Этап 2: шаблоны, недельные циклы, заполнение слайда

**Что делали:** реализовали Этап 2 по плану — CRUD шаблонов и недельных циклов для админа,
форму заполнения слайда с живым предпросмотром для спикера. Полный план решения (с
уточнёнными на старте вопросами по scope) сохранён в `C:\Users\mille\.claude\plans\lucky-petting-glacier.md`.

**Уточнено с пользователем перед реализацией:**
- Недельные циклы получили простой CRUD для админа (иначе `Slide.weeklyCycleId` физически нечего
  проставлять).
- В этом этапе НЕТ перехода `DRAFT → SUBMITTED` — только черновик + предпросмотр. Review-воркфлоу
  (`NEEDS_REVISION`, комментарии) — отдельный, следующий этап.
- Спикер видит все активные шаблоны, без фильтра по `isShared`/владельцу.

**Backend (миграция не нужна, схема не менялась):**
- `server/src/routes/weeklyCycles.ts` — GET (любой авторизованный), POST/PATCH (ADMIN).
- `server/src/routes/templates.ts` — GET список/один (любой авторизованный), POST с вложенными
  полями, PATCH с diff-синхронизацией полей (транзакция; удаление поля, уже использованного в
  `SlideFieldValue`, аккуратно даёт 409, а не 500 — проверено на практике, транзакция откатывается
  целиком).
- `server/src/routes/slides.ts` — GET только свои слайды; POST идемпотентен (get-or-create по
  `ownerId+weeklyCycleId+templateId`, поэтому отдельный экран «список моих слайдов» не нужен);
  PATCH пишет `FieldValueHistory` на каждое изменившееся значение, блокирует правку, если
  `slide.status !== "DRAFT"` или `weeklyCycle.status !== "COLLECTING"`.
- Подключены в `server/src/app.ts` под `/api/weekly-cycles`, `/api/templates`, `/api/slides`.

**Frontend:**
- `client/src/components/AppHeader.tsx` — общий хедер с ролевой навигацией (вынесен из
  `DashboardPage`).
- `client/src/pages/TemplatesPage.tsx`, `CyclesPage.tsx` (ADMIN), `SlideFormPage.tsx` (SPEAKER) —
  новые экраны, маршруты в `App.tsx` защищены `<ProtectedRoute role="...">`.
- `client/src/api/client.ts` — расширен методами для templates/cycles/slides.

**Найденный и исправленный баг (не в исходном плане):** `API_BASE` в `client/src/api/client.ts`
был жёстко прописан как `http://localhost:4000/api`. Поскольку в Этапе 1 клиент на этой машине
пришлось поднимать на `127.0.0.1:5173` (см. IPv6-нюанс vite в README), а не `localhost:5173`,
браузер считает `127.0.0.1` и `localhost` **разными сайтами** — `SameSite=Lax`-кука с токеном не
отправлялась на кросс-сайтовые запросы после логина. Логин при этом успешно отрабатывал (кука
только ставится), поэтому баг не проявился в проверке Этапа 1, где после входа не было
дополнительных authenticated-запросов, кроме тех, что рендерились из данных самого login-ответа.
Проявился только сейчас, когда `CyclesPage`/`TemplatesPage`/`SlideFormPage` стали делать
собственные GET-запросы после монтирования. **Исправлено**: `API_BASE` теперь берёт хост из
`window.location.hostname` вместо хардкода — работает одинаково что на `localhost`, что на
`127.0.0.1`. В проде (nginx same-origin по README) это ничего не меняет.

**Проверка:**
1. `npx tsc --noEmit`/`npx tsc -b` — чисто на сервере и клиенте.
2. curl-проверка под обеими ролями: создание цикла и шаблона админом, идемпотентный
   get-or-create слайда и сохранение значений спикером, 403 на попытку спикера дёрнуть
   `/api/templates`, 409 + откат транзакции при попытке удалить занятое поле шаблона, 403 при
   попытке редактировать слайд в `ARCHIVED`-цикле.
3. Playwright-прогон в браузере (после фикса SameSite-бага): вход админом → создание цикла и
   шаблона через UI → выход → вход тестовым спикером → заполнение слайда → живой предпросмотр
   обновляется по мере ввода → сохранение → **перезагрузка страницы → значения на месте** →
   попытка спикера открыть `/templates` напрямую по URL → редирект на `/`. Скриншоты сохранены
   локально в scratchpad сессии (в репозиторий не коммитились).

**Тестовые данные, оставленные в БД (не часть деливери, безопасно удалить позже):** пользователь
`qa.speaker.tmp` (создан через `POST /api/users`, одноразовый), цикл «2026-W29», шаблон «Отчёт по
проекту» — у этого шаблона кириллица в названии/полях **отображается квадратиками**: это артефакт
кодировки при передаче `-d` с кириллицей через Bash-тул на этой Windows-машине (не баг
приложения — шаблон «UI-тест шаблон», созданный прямо через браузер, отрендерился корректно).

**Не сделано / осталось на будущее:**
- Отправка слайда на проверку (`DRAFT → SUBMITTED`) и экран проверки у админа
  (`NEEDS_REVISION`, комментарии) — сознательно вынесено за рамки этого этапа.
- Сборка презентации (`Presentation`/`PresentationSlide`) — модели есть, экранов нет.

## 2026-07-18 — Проверка Этапа 1 end-to-end

служба (`postgresql-x64-17`, уже был запущен), Node.js v24.18.0.

**Обнаруженное состояние (было сделано в предыдущей сессии, до этого лога):**

- База `weekly_report` уже создана, миграция `20260715181059_init` уже применена — все 11 таблиц
  на месте (`User`, `Template`, `TemplateField`, `Slide`, `WeeklyCycle`, `Presentation`,
  `PresentationSlide`, `SlideFieldValue`, `FieldValueHistory`, `AuditLogEntry`,
  `_prisma_migrations`).
- Seed уже был выполнен: пользователи `admin` (роль ADMIN) и `speaker.demo` (роль SPEAKER),
  пароль админа из `.env` — `SEED_ADMIN_PASSWORD=ChangeMe123!`.
- `node_modules` и `package-lock.json` в обеих папках уже на месте.

**Что проверили в этой сессии:**
1. Подняли `npm run dev` в `server/` (порт 4000) — стартует, `/api/health` отвечает `{"ok":true}`.
2. `POST /api/auth/login` с admin/ChangeMe123! — возвращает 200, ставит httpOnly JWT-куку.
3. `GET /api/users` (защищён `requireAuth` + `requireRole("ADMIN")`) с этой кукой — возвращает
   список пользователей. Ролевая модель на бэкенде подтверждена рабочей.
4. Подняли `npm run dev` в `client/` (порт 5173), прогнали реальный вход через headless Chromium
   (Playwright): страница логина рендерится → ввод admin/ChangeMe123! → редирект на дашборд с
   именем «Администратор системы» и плашкой роли «Администратор». Скриншоты отправлены
   пользователю в чат (не сохранены в репозитории).

**Найденный нюанс (не баг кода, особенность этой машины):** dev-сервер `vite` по умолчанию
слушает `localhost` только на IPv6 (`[::1]`); внешние подключения (curl, Playwright, обычный
`Invoke-WebRequest`) до него не достучались, хотя `netstat` показывал `LISTENING`. Порт 4000
(Express) при этом слушал `0.0.0.0`/`[::]` и проблем не было — значит дело именно в дефолтном
хосте vite. Обошли через `npx vite --host 127.0.0.1` + синхронную правку
`CLIENT_ORIGIN="http://127.0.0.1:5173"` в `server/.env` (иначе CORS блокирует запросы с
фронтенда). Заметка добавлена в `README.md`. На боевом Linux-сервере такого не ожидается.

**Итог:** Этап 1 подтверждён рабочим end-to-end на этой машине. `README.md` обновлён — убрана
пометка «не проверялось», добавлена дата проверки и заметка про IPv6-нюанс vite.
