## 2026-07-18 — Этап 4: независимое тестирование (`tester`) нашло и мы исправили гонку, роняющую сервер

**Что сделали:** прогнали субагента `tester` независимо от реализации (каталог кейсов —
`TEST_CASES.md`, ~50 кейсов: интеграционные через Node `fetch`, e2e через Playwright). Первый
запуск оборвался по сетевой ошибке на середине, второй запуск возобновили тем же агентом через
`SendMessage` (сохранил контекст, не начинал заново).

**Найденные дефекты и исправления:**

1. **Критично — исправлено.** Гонка в `getOrCreatePresentation`
   (`server/src/routes/presentations.ts`): при одновременном первом добавлении слайда/заглушки
   в презентацию цикла (два админа/две вкладки) второй запрос падал на уникальном констрейнте
   `Presentation.weeklyCycleId` (`P2002`), и, так как в проекте нет глобального перехвата
   необработанных `reject`, это ронял весь Node-процесс (воспроизведено дважды, стектрейс был в
   `server.log`, все пользователи теряли доступ до перезапуска). Исправлено: `POST .../slides` и
   `POST .../placeholders` теперь оборачивают транзакцию в `try/catch`, при `P2002` возвращают
   чистый `409` вместо падения. Дополнительно в `server/src/index.ts` добавлен глобальный
   `process.on("unhandledRejection", ...)` как страховка на случай подобных гонок в других роутах
   (проблема системная — большинство роутов в проекте не оборачивают async-хендлеры в try/catch).
   Перепроверено вручную: два параллельных `curl` на новый цикл → один `201`, второй чистый `409`,
   сервер остался жив (`/api/health` отвечает).
2. **Средне — исправлено.** `PATCH .../order` не проверял длину входного массива, только
   совпадение множества id — `order` с дублирующимся id проходил как валидный и портил
   нумерацию `order` у слотов. Добавлена проверка `order.length === slides.length` вместе с
   проверкой на дубликаты. Перепроверено: массив с повторным id → `400`.
3. **Средне — вопрос к заказчику, не баг.** Просмотр презентации (и для админа, и для спикера)
   показывает только имя спикера + название шаблона + статус, но не сами значения полей слайда —
   по факту нельзя увидеть содержание доклада, только его наличие в списке. Это буквально
   соответствует одобренному плану, но `tester` справедливо предложил свериться с заказчиком,
   действительно ли это то, что имелось в виду под «просмотром презентации» (решение №8 в
   `REQUIREMENTS.md`). Оставлено на усмотрение следующей сессии/пользователя.
4. **Низкий приоритет — известное ограничение, не регрессия.** `ARCHIVED`-цикл не защищён от
   попытки собрать в него презентацию — но `ARCHIVED` явно припаркован вне рамок Этапа 4
   (см. `REQUIREMENTS.md`), фиксируем как заметку на будущее, не чиним сейчас.

Все специально запрошенные негативные кейсы (чужой цикл в `slideId` → 404, чужой/несуществующий
id в `order` → 400, повторное добавление уже `IN_PRESENTATION` слайда → 403, удаление
несуществующего/чужого слота → 404) прошли на первом запуске без доработок.

**Тестовые данные, оставшиеся в БД:** от `tester` — несколько `qa.*.stage4*`/`qa.race.repro*`/
`qa.archived.check*`/`qa.orderlen.check*` спикеров и циклов `2026-W-*` (в основном `ASSEMBLED`,
один `ARCHIVED` с презентацией — как улика для пункта 4). От перепроверки фиксов в этой сессии —
цикл «2026-W98 (race retest)» с одним слотом. Пароли тестовых спикеров не сохранены.

**Дополнительно по вопросу №3:** пользователь подтвердил — просмотр презентации должен показывать
содержимое слайда, не только имя/шаблон/статус. Добавлено: `includeSlideForSlot` в
`server/src/routes/presentations.ts` теперь включает `fieldValues`, `AssemblePage.tsx` рендерит
поля шаблона со значениями (пустые — прочерк), по образцу `ReviewPage.tsx`. Проверено Playwright:

**Найденный и исправленный баг (не в исходном плане):** `API_BASE` в `client/src/api/client.ts`
был жёстко прописан как `http://localhost:4000/api`. Поскольку в Этапе 1 клиент на этой машине
пришлось поднимать на `127.0.0.1:5173` (см. IPv6-нюанс vite в README), а не `localhost:5173`,
браузер считает `127.0.0.1` и `localhost` **разными сайтами** — `SameSite=Lax`-кука с токеном не
отправлялась на кросс-сайтовые запросы после логина. Логин при этом успешно отрабатывал (кука
только ставится), поэтому баг не проявился в проверке Этапа 1, где после входа не было
дополнительных authenticated-запросов, кроме тех, что рендерились из данных самого login-ответа.
Проявился только сейчас, когда `CyclesPage`/`TemplatesPage`/`SlideFormPage` стали делать
