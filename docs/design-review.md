## 2026-07-19 — Этап 5, шаг 1: варианты компоновки слайда презентации (только дизайн-предложение)

**Задача:** визуальный дизайн одного слайда презентации (не код, не схема) — предложить
2-4 варианта layout для будущего `Template`/`TemplateField`-подобного движка, взяв за основу
референс `референс общей презентации.pptx` (16 слайдов, изучены все).

**Что было (референс):** плоский список label→текст (`TemplateField.label` + `SlideFieldValue.value`),
рендерится дженерик-списком `label: value` в `ReviewPage.tsx`/`AssemblePage.tsx`/`SlideFormPage.tsx` —
без структуры, без графиков, без header/footer-специфики.

**Наблюдения по референсу (все 16 слайдов):**
- Слайд 1 — титульный (лого, название встречи, дата), не паттерн для контентных слайдов.
- Слайды 13, 15 — просто разделители-заглушки по отделам (имя отдела на белом фоне).
- Слайд 3 (Соколовская А.) — шапка-бар с именем, подзаголовок, 2 bar-чарта план/факт бок о бок,
  под ними мини-таблица план/факт-прогноз/% исполнения, футер — тонкая линия без headcount-инфо.
- Слайды 7, 9, 11, 14, 16 — шапка (отдел+ФИО[+неделя][+лого]), 2×2 (или 2-колоночный)
  грид секций «События/Метрики/Планы/Риски», футер-бэнд «Отпуска, больничные» с буллет-списком
  headcount (отпуск/больничный/командировка/найм/увольнение/вакансии).
- Слайд 10 (подрядчики) — самый насыщенный: несколько таблиц + line-chart + bar-chart как
  растровые embedded-картинки (это и есть два ~100KB media-файла из pptx, а не фото людей — фото
  в референсе вообще не используются, только лого компании (маленькая картинка) и два chart-скрина).
- Вывод: единого слайда «на все случаи» в референсе нет — есть де-факто 2 семейства: «квадрант
  из текстовых блоков» (большинство отделов) и «финансовый, с чартами» (продажи/финансы).

**Предложенные варианты (переданы пользователю для решения, mockup:
`slide-layout-variants.html` во временной директории агента — переслать при необходимости
пользователю отдельно, файл не сохранён в репозитории):**

1. **Вариант A — «Квадрант».** Header (отдел/ФИО/неделя/лого) + опциональная KPI-строка
   (1-3 metric-tile) + фиксированный грид 2×2 из rich-text-секций + footer-band headcount.
   Ближе всего к большинству слайдов референса, проще всего реализовать (Consta Grid/Card),
   но не годится для финансовых/чарт-слайдов без второго layoutKind.
2. **Вариант B — «Гибкий конструктор секций».** Header + KPI-строка + список типизированных
   секций произвольной длины (rich-text / table / chart-image), автор сам расставляет
   колонки/порядок + фиксированный footer-band. Покрывает и квадрант, и финансовые слайды одной
   схемой, но требует значительно больше UI-работы в `SlideFormPage.tsx` (типы блоков, загрузка
   графика/картинки, редактор таблиц) и рискует переусложнить live-preview.
3. **Вариант C — «Простой одноколоночный».** Header + опциональная KPI-строка + вертикальный
   список секций любого типа + footer-band. Самый дешёвый в реализации и никогда не «ломается»
   от переполнения текста, но хуже соответствует референсу визуально (компактность/обозримость
   двухколоночных слайдов теряется).
4. **(в рамках B/A) «Несколько layoutKind по Template»** — не отдельный вариант компоновки, а
   архитектурный приём: раз в приложении уже есть сущность `Template` на роль/отдел, логично
   каждому `Template` присвоить `layoutKind` (квадрант / финансовый-чартовый / простой) вместо
   одного максимально универсального движка — меньше сложности при той же гибкости, которая
   реально нужна (в референсе только 2 семейства).

**Необходимые типы блоков (для будущей схемы, не реализовано):** `metric-tile` (label+value,
опц. план/факт/%), `rich-text-section` (label + буллет/абзацный текст), `chart-image`
(загрузка изображения графика или, в будущем, embed структурированных данных для чарта),
`table` (произвольные строки/колонки план-факт), `footer-stats` (headcount: отпуск/больничный/
командировка/найм/увольнение — по сути тот же rich-text с фиксированной семантикой и местом).

**Что нарочно не сделано:** ни схема (`schema.prisma`), ни `SlideFormPage.tsx`/`ReviewPage.tsx`/
`AssemblePage.tsx` не менялись — это визуальное предложение на обсуждение, ждём решения
пользователя по варианту(ам) прежде чем проектировать миграцию `TemplateField` → типизированные
блоки. Consta ещё не установлена (`@consta/uikit` не добавлен в `client/package.json`) — при
переходе к реализации потребуется явное разрешение на `npm install @consta/uikit` (см. общую
инструкцию агента `ux-ui-designer`).
