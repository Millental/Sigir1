## 2026-07-19 — Этап 5: ответы на открытые вопросы аналитика — все 9 закрыты, объём вырос

**Статус:** решения приняты пользователем, план ещё не составлен, код/схема не менялись. Дополняет
запись ниже (тот же день) — там был список вопросов, здесь — ответы на них.

1. **Слоты блоков по `layoutKind`** — гибко: `layoutKind` задаёт только визуальный
   каркас-контейнер, состав/число блоков внутри него можно менять. Значит нужен настоящий
   конструктор блоков, а не просто фиксированная форма на layoutKind.
2. **Миграция существующих данных — выросла в отдельную постоянную фичу.** Не разовый скрипт
   миграции 2 dev-шаблонов, а **загрузчик легаси-презентаций как постоянная возможность
   приложения**: админ загружает старый pptx (в т.ч. для онбординга новых отделов в будущем),
   система разбирает файл, конвертирует в новую блочную модель и **предлагает** вариант
   компоновки (`layoutKind` + блоки) — админ смотрит предложение и утверждает/правит его в
   конструкторе (см. п.3). Встроенные в pptx растровые графики/картинки при разборе
   автоматически становятся блоками `chart-image`.
3. **Кто редактирует `layoutKind`/блоки** — нужен **полноценный UI-конструктор шаблонов** в
   Этапе 5 (создание шаблона с нуля вручную ИЛИ через импорт pptx с последующей правкой
   предложенного варианта), не заглушка/захардкоженный набор.
4. **`chart-image` и импортёр требуют файлового хранилища** — **строим upload-инфраструктуру с
   нуля** (в проекте её сегодня нет: ни `multer`, ни хранилища). Нужна и для ручной загрузки
   картинки графика на слайд, и для приёма pptx-файла легаси-импортёром.
5. **`footer-stats`** — подтверждено: чистый ручной ввод, никакой интеграции с HR-системой в
   этом этапе.
6. **Live-preview / UX редактирования** — **разделены два разных экрана с разным уровнем
   сложности**: полноценный конструктор блоков (drag/drop, ручное добавление/правка, приём
   предложения импортёра) — это экран **администратора для работы с шаблоном**; форма
   заполнения слайда **спикером** (`SlideFormPage.tsx`) остаётся простой — блок за блоком,
   без перетаскивания, спикер просто вводит значения в уже готовую структуру блоков шаблона.
7. **Неизменность `layoutKind`/структуры блоков** — замораживаются, как только по шаблону
   существует хотя бы один слайд (тот же паттерн, что уже есть для `TemplateField`
   сегодня — `409` при попытке тронуть занятое поле).
8. **Объём фикса видимости для спикера** — точечная фильтрация в двух уже найденных местах
   (`GET /api/weekly-cycles`, `GET /api/presentations/cycle/:id`), отдельный per-role эндпоинт
   не нужен.
9. **Экран «список презентаций» для архивации** — не новый экран, **расширяем существующий
   `/cycles`** (добавить признак «презентация собрана» + действие «в архив»).

**Вывод: объём Этапа 5 существенно вырос против первоначального черновика** — помимо
архивации и редизайна слайда, добавились: (а) полноценный конструктор шаблонов с блоками,
(б) файловое хранилище с нуля, (в) парсер/импортёр legacy pptx с эвристикой автоматического
разбора на блоки и предложением компоновки. Каждый из пунктов (а)-(в) по сложности сопоставим
с отдельным этапом сам по себе — стоит обсудить с пользователем, разбивать ли Этап 5 на
под-этапы (например: 5a — архивация + фикс видимости; 5b — блочная модель + конструктор шаблонов
вручную; 5c — файловое хранилище + legacy-импортёр) при переходе в план-режим, а не пытаться
реализовать всё одним планом.

## 2026-07-19 — Этап 5: сбор требований (архивация цикла + редизайн компоновки слайда) — реализация ещё не начата

**Статус:** черновик для уточнения с заказчиком, по образцу записи Этапа 4 ниже. Кода, миграций и
изменений схемы в этой сессии не делалось — только разбор фактического состояния кода/БД и список
вопросов. Часть решений по Этапу 5 уже принята в предыдущих сессиях (обсуждение scope 2026-07-19,
дизайн-предложение `ux-ui-designer` в `docs/design-review.md`) — здесь они формализуются как
требования, не пересматриваются заново.

### Что уточнили (разбор фактического кода и БД, не пересказ задачи)

1. **Текущая модель слайда полностью плоская**: `Template` → `TemplateField` (`label`,
   `isRequired`, `order`) → `SlideFieldValue` (`value: String`), см. `server/prisma/schema.prisma`.
   Поля `layoutKind` и типизированных блоков (`metric-tile`/`rich-text-section`/`chart-image`/
   `table`/`footer-stats`) в схеме сегодня нет — их ещё предстоит спроектировать.
2. **В проекте нет инфраструктуры загрузки файлов**: в `server/src` нет ни `multer`, ни любого
   другого upload-миддлварá, `package.json` сервера не содержит соответствующих зависимостей.
   Это прямо касается `chart-image` — сегодня негде физически хранить загруженную картинку.
3. **Видимость циклов сегодня не фильтруется вообще.** `GET /api/weekly-cycles`
   (`server/src/routes/weeklyCycles.ts`) отдаёт **все** циклы **любому** авторизованному
   пользователю без учёта статуса и роли. `AssemblePage.tsx` (клиент) использует этот список
   напрямую для выпадающего списка на `/presentation` **без какой-либо фильтрации** — и для
   админа, и для спикера. `SlideFormPage.tsx`, для сравнения, уже фильтрует список на клиенте до
   `status === "COLLECTING"` для своей формы заполнения — то есть фильтрация «что видно» решается
   сегодня по месту, а не централизованно.
4. **`GET /api/presentations/cycle/:weeklyCycleId`** (`server/src/routes/presentations.ts`) не
   имеет ролевого ограничения на чтение и не смотрит на `weeklyCycle.status` — отдаёт презентацию
   цикла (если собрана) с полным содержимым слайдов любому авторизованному пользователю для
   любого id цикла, включая уже `ARCHIVED`. Единственное, что ограничено ролью ADMIN — список
   `candidateSlides` (кандидаты на добавление).
5. **Из пунктов 3–4 следует: правило «спикер видит только собранные готовые презентации, никогда
   черновики и никогда архивные» — это НЕ чисто аддитивная доработка.** Технически сегодня
   спикер может выбрать в выпадающем списке на `/presentation` любой цикл, включая уже
   существующий в dev-БД архивный («2026-W29» — см. п.6), и получить его презентацию через тот же
   `GET .../cycle/:id`, который не проверяет ни роль, ни статус. Понадобится фильтрация минимум в
   двух местах: список циклов, предлагаемых спикеру для выбора, и сам эндпоинт чтения презентации.
6. **Утечки содержимого черновиков сегодня нет** — `candidateSlides` (сырые `SUBMITTED`-слайды)
   отдаются только ADMIN, а содержимое `PresentationSlide` для любой роли — это всегда уже
   реально собранные (`SUBMITTED`/`IN_PRESENTATION`) слайды, не черновики других спикеров. Пробел
   именно в видимости на уровне цикла/презентации целиком (в т.ч. архивной), не в утечке
   недоделанного контента отдельного спикера.
7. **`CyclesPage.tsx`** (админ-экран управления циклами) уже содержит лейбл статуса «Архив» и
   позволяет через существующий свободный `PATCH /api/weekly-cycles/:id` выставить циклу
   `ARCHIVED` вручную, без проверки, что презентация вообще существует и собрана — это и есть
   известное ограничение, отмеченное `tester`-агентом в `SESSION_LOG.md` (пункт 4 записи Этапа 4).
   Отдельного экрана «список всех презентаций» в проекте сегодня нет — есть только
   выбор-одного-цикла-из-списка на `/presentation` (`AssemblePage.tsx`) и общий список циклов на
   `/cycles`.
8. **Реальное состояние dev-БД проверено запросом (не предположение):** 2 шаблона («Отчёт по
   проекту» — 3 поля, 7 слайдов используют; «UI-тест шаблон» — 2 поля, 11 слайдов используют),
   18 слайдов всего, 42 значения полей всего, 15 недельных циклов в статусах `COLLECTING`/
   `ASSEMBLED`/`ARCHIVED` (в т.ч. один уже реально `ARCHIVED` — «2026-W29», оставленный тестовыми
   данными Этапа 4, — то есть это состояние не гипотетическое, оно уже существует в БД).

### Принятые решения (формализация уже согласованного в предыдущих сессиях)

1. **Архивация — только из уже собранной (`ASSEMBLED`) презентации, только ADMIN.** Обоснование:
   продолжает уже устоявшийся в проекте паттерн «все переходы жизненного цикла — админ-only»
   (шаблоны, циклы, `/review`, сборка презентации) и логически привязывает `ARCHIVED` к наличию
   реального артефакта, а не к произвольному статусу цикла без презентации.
2. **Видимость: `ARCHIVED`-презентация видна только админу; спикеры видят в любом списке только
   собранные готовые презентации — никогда черновики, никогда архивные.** Обоснование: явное
   подтверждение пользователя в сессии 2026-07-19 (см. `SESSION_LOG.md`).
3. **Экспорт/PDF — снова отложен, вне рамок Этапа 5.** Обоснование: решение по-прежнему зависит
   от того, как в итоге будет выглядеть просмотр/пролистывание презентации — пользователь явно
   не готов фиксировать объём экспорта раньше этого.
4. **Каждый `Template` получает поле `layoutKind`** (ориентировочные значения: квадрант /
   финансовый-чартовый / простой одноколоночный) вместо одной универсальной компоновки слайда на
   все случаи. Обоснование: пользователь подтвердил этот архитектурный приём после того, как
   `ux-ui-designer` разобрал все 16 слайдов реального референса (`референс общей презентации.pptx`)
   и показал, что там всего 2 реальных семейства слайдов + fallback — не нужна максимальная
   гибкость одного мега-движка, достаточно небольшого набора компоновок на выбор.
5. **Набор типов блоков для будущей модели:** `metric-tile` (label+value, опц. план/факт/%),
   `rich-text-section` (label + буллет/абзацный текст), `chart-image` (загруженное изображение
   графика), `table` (произвольные строки/колонки, напр. план/факт по месяцам), `footer-stats`
   (headcount по отделу: отпуска/больничные/командировки/найм/увольнения/вакансии — семантически
   тот же rich-text, но с фиксированным местом на слайде). Обоснование: список выведен напрямую из
   разбора всех 16 слайдов референса, зафиксирован в `docs/design-review.md`.

### Открытые вопросы (нужен ответ до планирования)

1. **Как `layoutKind` определяет доступный набор блоков для шаблона** — это фиксированный набор
   слотов на каждый `layoutKind` (например, «квадрант» = ровно 4 ячейки `rich-text-section` +
   опциональная строка `metric-tile` + фиксированный `footer-stats`, без права что-то добавить или
   убрать), или внутри каждого `layoutKind` всё ещё остаётся гибкость (можно менять число/состав
   блоков, `layoutKind` только задаёт визуальный каркас-контейнер)? От ответа зависит, придётся ли
   вообще строить конструктор блоков для админа, или достаточно фиксированных форм на layoutKind.
2. **Путь миграции существующих данных Этапов 2–4.** В dev-БД реально есть 2 шаблона (7 и 11
   слайдов соответственно, 42 значения полей всего — цифры проверены запросом, не предположение).
   Получают ли они дефолтный `layoutKind` (например, «простой одноколоночный») с автоматическим
   отображением существующих `TemplateField`/`SlideFieldValue` на новые типизированные блоки (во
   что именно — в `rich-text-section`, видимо, по умолчанию)? Или эти dev-данные считаются
   расходным материалом и их можно не мигрировать (пересоздать шаблоны с нуля после того, как
   новая модель заработает)? Нужно явное решение, а не предположение по умолчанию.
3. **Кто редактирует `layoutKind` и структуру блоков шаблона** — это часть Этапа 5 (полноценный
   UI авторинга шаблонов для админа: выбор `layoutKind`, настройка блоков), или для первого среза
   достаточно захардкодить/засеять небольшой фиксированный набор шаблонов с готовой раскладкой, а
   полноценный UI-конструктор оставить на более позднему этапу?
4. **`chart-image` — что это технически.** Обычная загрузка файла (PNG/JPG) с последующим
   хранением — если да, то где физически хранить файлы (в проекте сегодня нет ни файлового
   хранилища, ни upload-миддлварá — ни `multer`, ни аналогов, проверено по `server/src` и
   `package.json`)? Или ожидается какой-то другой способ разместить картинку графика на слайде
   (вставка по URL, base64 в самом значении поля, что-то ещё)? Это прямо влияет на объём Этапа 5 —
   файловое хранилище с нуля существенно дороже, чем текстовое поле.
5. **`footer-stats` — подтвердить, что это ручной ввод**, как и все остальные поля сегодня (то
   есть спикер/админ печатает цифры руками), а не интеграция с внешней HR-системой сейчас. Судя по
   обсуждению, это предполагается «просто ещё один блок с фиксированным местом», но явное
   подтверждение для Этапа 5 нужно, чтобы не закладывать интеграцию, которая не нужна.
6. **Живой предпросмотр в `SlideFormPage.tsx`** — должен продолжать работать в целом как сейчас
   (по одному блоку-типу за раз, просто больше типов блоков вместо универсального `label: value`),
   или в объём Этапа 5 входит более крупный пересмотр UX редактирования слайда (например, drag/
   drop блоков, визуальный редактор раскладки)? Разница в объёме работ значительная.
7. **Меняется ли `layoutKind` после того, как по шаблону уже есть слайды** — выбирается один раз
   при создании шаблона и дальше неизменен (по аналогии с тем, как сегодня в проекте обращаются с
   изменением полей уже использованного шаблона — `PATCH` даёт `409`, если поле занято значением),
   или допускается смена `layoutKind` в любой момент с риском «разъехавшихся» существующих
   значений блоков?
8. **Найденный при разборе кода конфликт с текущим поведением (см. «Что уточнили», пп.3–5):
   правило видимости для спикера — это фильтрация в уже существующих местах, или нужен более
   широкий пересмотр?** Сегодня `GET /api/weekly-cycles` не фильтрует вообще ничего, а
   `GET /api/presentations/cycle/:id` не проверяет ни роль, ни статус цикла при чтении. Значит
   правило «спикер не видит архивные/черновые» нужно явно реализовать хотя бы в двух местах
   (список циклов, предлагаемый спикеру для выбора на `/presentation`; сам эндпоинт чтения
   презентации) — подтвердите, что это ожидаемый объём, а не что-то более широкое (например,
   отдельный per-role эндпоинт списка вместо доработки текущего общего).
9. **Нужен ли отдельный экран «список всех презентаций»** — формулировка решения №1
   («администратор переводит презентацию в архив из списка всех презентаций») подразумевает некий
   список, но сегодня в проекте нет отдельного экрана-списка презентаций: есть общий список циклов
   на `/cycles` (админ, с уже существующим лейблом «Архив», но без привязки к наличию собранной
   презентации) и выбор-одного-цикла-из-выпадающего-списка на `/presentation`. Строить новый
   экран-список (циклы, у которых `presentation` не `null`, с кнопкой «В архив»), или расширить
   один из существующих экранов?

### Предлагаемый объём работ (черновик, зависит от ответов выше — не финальный план)

Ориентировочно, без привязки к конкретным файлам (будет уточнено в плане после ответов):

**Архивация:**
- Новый (или доработанный существующий) эндпоинт перевода цикла в `ARCHIVED`, с проверкой на
  бэкенде, что `presentation` для цикла существует (не просто `PATCH` статуса вслепую, как сейчас).
- Фильтрация видимости для роли SPEAKER — минимум в списке циклов/презентаций, предлагаемом на
  `/presentation`, и в самом чтении презентации по id.
- Экран или доработка экрана для просмотра списка презентаций с возможностью архивации (объём —
  вопрос №9).

**Компоновка слайда:**
- Миграция схемы: `layoutKind` на `Template`, новая модель типизированных блоков вместо (или
  вместе с) `TemplateField`/`SlideFieldValue` — точная форма зависит от ответа на вопрос №1.
- Возможно новая модель/поле для хранения загруженных изображений (`chart-image`) — зависит от
  ответа на вопрос №4; если нужна загрузка файлов, это первое появление файлового хранилища в
  проекте и отдельный, не тривиальный кусок работы (нет прецедента в кодовой базе).
- Доработка `SlideFormPage.tsx` (форма + предпросмотр), `ReviewPage.tsx`, `AssemblePage.tsx` под
  рендер типизированных блоков вместо генерик `label: value` списка — объём зависит от ответа на
  вопрос №6.
- Миграция существующих 2 шаблонов / 18 слайдов / 42 значений полей — объём зависит от ответа на
  вопрос №2.

## 2026-07-18 — Этап 4: ответы заказчика на открытые вопросы (реализация ещё не начата)

Ответы получены на все 9 вопросов из предыдущей записи. Ниже — решения, зафиксированные для

1. **Критерий готовности — вариант (а).** Любой слайд в статусе `SUBMITTED` подходит для включения
   в презентацию. «Принять» на `/review` остаётся информационной отметкой в журнале аудита; реальным
   «одобрением» становится сам факт выбора админом при сборке. Решение Этапа 3 (audit-log-only
   approve) не пересматривается.
2. **Сборка переводит слайд в `IN_PRESENTATION`, и это блокирует дальнейшее редактирование**, пока
   слайд входит в презентацию. При исключении слайда из состава (частичная правка, см. п.3) или при
   полной «разборке» презентации статус **откатывается обратно в `SUBMITTED`** (не остаётся
   историческим следом).
3. **Нужно частичное редактирование уже собранной презентации** — добавить/убрать отдельный слайд,
   поменять порядок, без полного пересоздания записи `Presentation`. Значит: `POST` создаёт
   презентацию цикла один раз, дальнейшие изменения состава/порядка идут через отдельный
   `PATCH`-подобный эндпоинт, который трогает только затронутые `PresentationSlide`
   (добавление создаёт запись + `IN_PRESENTATION`, удаление удаляет запись + откат в `SUBMITTED`,
   на `weeklyCycleId` по-прежнему висит один `Presentation` на цикл — редактируется, не пересоздаётся).
4. **Ручной выбор и ручной порядок**, управление — стрелки вверх/вниз (без drag-and-drop).
5. **Частичная сборка «с кем готово»** — не нужно ждать, пока все слайды цикла станут `SUBMITTED`.
   Дополнительно: **должна быть возможность поставить «заглушку»** вместо слайда спикера, который не
   успел подготовиться, чтобы презентация не имела пропуски по составу спикеров. Механизм заглушки
   не до конца специфицирован ответом — нужно решить в план-режиме: это (i) отдельный
   `PresentationSlide` без привязки к реальному `Slide` (текстовая заглушка «доклад не готов» +
   имя спикера), или (ii) обычный `Slide`, который админ сам создаёт/помечает как заглушку от имени
   отсутствующего спикера. Зафиксировано как техническая деталь для уточнения на этапе планирования,
   не как открытый вопрос заказчику — общее назначение (визуальный place-holder за не успевшего
   спикера) понятно.
6. **Сборка должна сама переключать `WeeklyCycle.status` в `ASSEMBLED`**, замораживая цикл для
   дальнейшего редактирования спикерами (используя уже существующую проверку

7. **PDF/печать/экспорт — не в этом этапе.** Отложено до одного из финальных этапов. Зафиксировано
   в разделе «НЕ входит в этап» ниже без изменений.
8. **Видимость: все авторизованные пользователи видят собранную презентацию.** Админ может
   управлять составом/порядком (редактирование, см. п.3, п.4). Спикеры — только просмотр

9. **`ARCHIVED` — не в объёме Этапа 4.** Явно **припарковано** как отдельное будущее требование:

   в `README.md`/бэклоге как известный будущий шаг, чтобы не потерялось.

### Что уточнили (разбор фактического кода)

Проверил модели, роуты и экраны напрямую, не полагаясь на пересказ задачи:

1. **`SlideStatus` уже содержит `IN_PRESENTATION`** (`server/prisma/schema.prisma`, зафиксировано
   ещё в миграции `20260715181059_init`), и это значение уже есть в маппинге лейблов на фронте

   присваивается. Это явный сигнал изначального замысла: сборка презентации должна переводить
   статус вошедших слайдов в `IN_PRESENTATION`. Нужно подтвердить.
2. **Кнопка «Принять» на `/review` не меняет `slide.status`** (`server/src/routes/slides.ts`,
   `PATCH /:id/review`, `action: "approve"`) — пишет только `AuditLogEntry` с
   `action: "SLIDE_APPROVE"`. Это осознанное решение Этапа 3 (см. `SESSION_LOG.md`), не баг. Но
   для Этапа 4 это значит: **в модели данных сейчас физически нет поля, которое отличало бы
   «слайд проверен и одобрен» от «слайд просто отправлен на проверку и ещё не смотрели»** — оба
   состояния выглядят как `SUBMITTED`. Единственный след «одобрения» — запись в
   `AuditLogEntry`, не привязанная жёстко к тому, что после неё не было повторного возврата на
   доработку. Это ключевая развилка для правил сборки (см. открытый вопрос №1).
3. **`Presentation.weeklyCycleId` помечено `@unique`** → у цикла максимум одна презентация в
   текущей модели. **`PresentationSlide.slideId` тоже `@unique`** → один слайд не может входить
   более чем в одну презентацию одновременно (а с учётом каскадного удаления `PresentationSlide`
   при удалении `Presentation` — переслайд возможен только через замену/удаление всей записи
   `Presentation`, не через `update`). Нужно подтвердить, что означает повторная сборка того же
   цикла (см. открытый вопрос №3).
4. **`WeeklyCycle.status`** (`COLLECTING → ASSEMBLED → ARCHIVED`) уже существует и меняется
   свободным `PATCH /api/weekly-cycles/:id` админом (`server/src/routes/weeklyCycles.ts`), без
   какой-либо связи с наличием реальной `Presentation`. Сегодня можно вручную выставить циклу
   статус `ASSEMBLED`, не создав ни одной презентации, и наоборот. Нужно решить, должна ли сборка
   презентации сама переключать статус цикла (см. открытый вопрос №6).
5. **Роутов и экранов под `Presentation`/`PresentationSlide` нет вообще** — проверено grep'ом по
   `server/src` (нет `presentations.ts`) и по `client/src/App.tsx` (нет соответствующего route).
   Это полностью новый функционал, не доработка существующего.
6. Готовая точка опоры: `GET /api/slides/cycle/:weeklyCycleId` (админ-only, уже есть с Этапа 3,
   возвращает слайды цикла со значениями полей, отсортированные по спикеру/шаблону) — годится
   как источник списка кандидатов на включение в презентацию без нового эндпоинта на чтение
   слайдов.
7. Просмотр списка циклов (`GET /api/weekly-cycles`) сейчас доступен **любому авторизованному**
   пользователю, не только админу (уже используется в `SlideFormPage` для выбора цикла спикером)
   — значит технической преграды к тому, чтобы спикер видел цикл, для которого собрана
   презентация, сегодня нет; вопрос только в том, нужен ли ему экран просмотра (см. вопрос №8).

### Открытые вопросы (нужен ответ до начала реализации)

   - (а) Любой слайд в статусе `SUBMITTED` — «Принять» на `/review` тогда чисто информационная

### Предлагаемый объём работ (черновик, зависит от ответов выше — не финальный план)

Ориентировочная раскладка для сессии в режиме плана, когда вопросы будут закрыты:

**Backend:**
- Новый файл `server/src/routes/presentations.ts`, подключить в `server/src/app.ts` под
  `/api/presentations`.
- `GET /api/presentations/cycle/:weeklyCycleId` — вернуть текущую презентацию цикла (если есть)
  со слайдами в порядке `order`, включая значения полей (переиспользовать форму `includeSlide` из
  `slides.ts`). Доступ — минимум ADMIN, возможно + сам спикер на свои слайды (вопрос №8).
- `POST /api/presentations` (`{ weeklyCycleId, slideIds: string[] }`, ADMIN-only) — валидирует на
  бэкенде, что каждый `slideId` принадлежит указанному циклу и удовлетворяет критерию готовности
  (не доверять слепо выбору с клиента — критерий зависит от ответа на вопрос №1), в транзакции
  создаёт/заменяет `Presentation` + `PresentationSlide` с проставленным `order`, переводит статус
  включённых слайдов (вопрос №2), пишет `AuditLogEntry` (например `PRESENTATION_ASSEMBLE`).
  Точная семантика «создать vs заменить» — из ответа на вопрос №3.
- Возможно `DELETE /api/presentations/:id` («разобрать» презентацию, откатить статусы слайдов) —
  нужен только если ответ на вопрос №3 предполагает явную разборку перед пересборкой.

**Frontend:**
- `client/src/pages/AssemblePage.tsx` (роут `/assemble`, `ProtectedRoute role="ADMIN"`) — выбор
  цикла (как в `ReviewPage`), список кандидатов из `GET /api/slides/cycle/:id` с отметкой
  доступности по критерию из вопроса №1, чекбоксы/порядок (форма ввода зависит от вопроса №4),
  кнопка «Собрать презентацию».
- `client/src/pages/PresentationViewPage.tsx` (роут `/presentation/:cycleId` или похожий) —
  read-only просмотр собранной презентации в порядке слайдов, без управляющих элементов.
  Доступность роли — по вопросу №8.
- `client/src/api/client.ts` — новые типы (`Presentation`, `PresentationSlide`) и методы
  (`getPresentation`, `assemblePresentation`, при необходимости `unassemblePresentation`).
- `client/src/components/AppHeader.tsx` — пункт(ы) навигации на новые роуты, по образцу уже
  существующих ролевых ссылок.

**Критерии приёмки (черновик, требует финального подтверждения после ответов на вопросы):**
- Админ может для выбранного недельного цикла увидеть подходящие слайды и одним действием собрать
  их в упорядоченную презентацию.
- Собранные слайды получают статус «В презентации» везде, где статус отображается (лейбл уже есть
  в интерфейсе, сейчас просто не используется).
- Собранную презентацию можно посмотреть в исходном выбранном порядке в read-only режиме.
- Повторный запуск сборки для того же цикла ведёт себя предсказуемо и без ошибки 500 из-за
  уникальности `weeklyCycleId` — конкретное поведение см. вопрос №3.
- Слайды, не попавшие в презентацию, не меняются.
- Не-админ не может инициировать сборку (403) — по аналогии со всеми остальными
  административными действиями проекта.

### Сводка для заказчика (проектный объём, ещё НЕ реализовано — предложение на согласование)

Это не отчёт о сделанной работе, а описание того, что предлагается сделать на Этапе 4, чтобы
согласовать объём до начала программирования.

**Что появится, если этап пойдёт в этой раскладке:**
- Администратор сможет выбрать недельный цикл, увидеть слайды, готовые к включению, и собрать их
  в единую упорядоченную презентацию одним действием — вместо того чтобы вручную сверять статусы
  по экрану проверки.
- И администратор (и, возможно, спикеры — зависит от ответа на вопрос №8) смогут посмотреть
  итоговый собранный вид презентации в веб-интерфейсе.
- Слайды, вошедшие в презентацию, будут явно помечены статусом «В презентации» — этот лейбл уже
  есть в интерфейсе с прошлых этапов, просто пока никогда не появляется.

**Что точно НЕ входит в этот этап (если не будет сказано иначе):**
- Экспорт в PDF, печать или любой формат для показа вне браузера — инфраструктуры для этого в
  проекте нет; можно добавить в объём, но это отдельная, не тривиальная задача (см. вопрос №7).
- Изменение решения Этапа 3 про статус `SUBMITTED`/«Принять» — если не будет явно запрошено
  (см. вопрос №1, вариант (в)).
- Перевод цикла в статус `ARCHIVED` — если не будет подтверждено как часть этого этапа
  (см. вопрос №9).

**Ключевые риски/ограничения, которые нужно знать заранее:**
- Модель данных в текущем виде допускает **только одну презентацию на цикл** — если позже
  понадобится хранить историю нескольких версий сборки одного цикла, потребуется менять схему
  и делать новую миграцию, а не просто дописывать код.
- Сегодня в системе нет надёжного признака «слайд одобрен» — кнопка «Принять» не меняет статус
  слайда, это сознательный выбор прошлого этапа. Этап 4 не может обойти это молча: либо критерий
  готовности к сборке будет более простым, чем ожидается («всё, что отправлено на проверку»),
  либо потребуется дополнительная работа с журналом аудита, либо нужно будет пересмотреть решение
  Этапа 3. Все три варианта имеют разную стоимость реализации — см. открытый вопрос №1.
